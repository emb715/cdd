# Session Timeout Strategy for Mobile Browsers

> **Generated by:** `/cdd:decide` (multi-agent decision planning)
> **Context:** Mobile users getting logged out after 5 minutes due to iOS timer suspension

---

## Problem Statement

**Current situation:**
- Token refresh uses 15-second timer
- iOS Safari suspends timers when tab backgrounded
- Users get logged out while still "using" the app (backgrounded)
- Desktop works fine (timers not suspended)

**Requirements:**
- Keep users logged in during normal mobile usage
- Respect security policy (max 30min token lifetime)
- Minimize battery/network overhead
- Support iOS Safari, mobile Chrome, Android

---

## Options Researched

### Option 1: Long-Lived Refresh Tokens

**Advocate Agent Research:**

**Pros:**
- Standard OAuth 2.0 pattern
- Minimal network overhead (refresh only when needed)
- Works across all browsers
- Well-supported by auth libraries

**Cons:**
- Violates security policy (30min token max)
- Higher security risk if token stolen
- Doesn't actually solve the "timer suspension" problem
- Requires backend changes

**Codebase Analysis:**
- Currently using short-lived access tokens (30min)
- No refresh token infrastructure
- Would require database schema changes
- Security audit required

**Recommendation:** ❌ Rejected - violates security policy

---

### Option 2: Heartbeat Ping Service

**Advocate Agent Research:**

**Pros:**
- Works around timer suspension (uses Page Visibility API)
- Detects actual user activity
- Minimal overhead (1 ping/min = <100 bytes)
- No backend changes (use existing session endpoint)
- Respects security policy

**Cons:**
- Extra network calls (battery impact)
- Adds complexity (new service)
- Requires proper background state detection
- Needs monitoring to avoid false positives

**Codebase Analysis:**
- Page Visibility API supported (see `src/utils/`)
- Session refresh endpoint exists (`/api/auth/refresh`)
- Pattern matches existing polling services
- TypeScript service pattern in `src/auth/`

**Recommendation:** ✅ Recommended - best fit for requirements

---

### Option 3: WebSocket Keepalive

**Advocate Agent Research:**

**Pros:**
- Real-time connection (no polling)
- Can detect disconnects immediately
- Standard for chat apps, multiplayer games

**Cons:**
- Massive overkill for this use case
- High complexity (WebSocket infrastructure)
- Battery drain (persistent connection)
- Network overhead (>1KB/min)
- Doesn't solve timer issue (still needs heartbeat)

**Codebase Analysis:**
- No WebSocket infrastructure currently
- Would require server changes
- Over-engineered for session management
- Pattern mismatch (app not real-time)

**Recommendation:** ❌ Rejected - overkill, high overhead

---

### Option 4: LocalStorage Fallback

**Advocate Agent Research:**

**Pros:**
- Client-side only (no network calls)
- Fast (no latency)
- Works offline

**Cons:**
- Doesn't solve the root problem (timer still suspended)
- Security risk (XSS can steal tokens)
- localStorage not reliable (can be cleared)
- Violates security best practices

**Codebase Analysis:**
- Security policy forbids storing tokens in localStorage
- Existing token storage uses httpOnly cookies
- Would require security audit approval

**Recommendation:** ❌ Rejected - security violation

---

## Analysis Agent Summary

**Objective Comparison:**

| Criteria | Long-Lived Tokens | Heartbeat Ping | WebSocket | LocalStorage |
|----------|-------------------|----------------|-----------|--------------|
| **Respects Security Policy** | ❌ No | ✅ Yes | ✅ Yes | ❌ No |
| **Solves Timer Suspension** | ❌ No | ✅ Yes | ✅ Yes | ❌ No |
| **Low Network Overhead** | ✅ Yes | ✅ Yes | ❌ No | ✅ Yes |
| **Low Complexity** | ⚠️ Medium | ✅ Yes | ❌ High | ✅ Yes |
| **Battery Friendly** | ✅ Yes | ✅ Yes | ❌ No | ✅ Yes |
| **No Backend Changes** | ❌ No | ✅ Yes | ❌ No | ✅ Yes |

**Recommendation:** **Option 2: Heartbeat Ping Service**

**Rationale:**
- Only option that solves the problem AND respects constraints
- Low overhead (estimated 0.3KB/min)
- Fits existing architecture patterns
- Can be implemented quickly (2-3 days)
- Monitoring can catch issues early

**Trade-offs:**
- Extra network calls (mitigated by 60s interval)
- New service to maintain (acceptable complexity)
- Battery impact (minimal at 60s interval)

---

## Final Decision

**Your Decision:** Implement **Heartbeat Ping Service** (Option 2)

**Implementation Plan:**
1. Create HeartbeatService with 60s interval
2. Implement Page Visibility API wrapper
3. Pause heartbeat when tab backgrounded (battery saving)
4. Resume heartbeat when tab foregrounded
5. Add monitoring for ping failures
6. Deploy to staging, test on iOS devices
7. Monitor for 48h before production

**Success Metrics:**
- Zero timeout issues on mobile (target)
- Network overhead < 1KB/min (requirement)
- Battery impact < 3% over 1 hour (acceptable)
- No performance regression on desktop

**Follow-up:**
- Monitor auth metrics for 1 week post-launch
- Adjust interval if battery impact higher than expected
- Consider adding user preference (disable heartbeat for ultra-low data mode)
